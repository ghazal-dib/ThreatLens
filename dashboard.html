Rule-Based Alerts

Five explicit, explainable rules built from known attacker behaviors.

How severity is decided
High = the behavior is dangerous on its own, with no extra context needed. Medium = the behavior is unusual but only meaningful combined with other evidence.
HIGH
Remote + elevated logons (lateral movement), anti-forensics tools (e.g. sdelete), and obfuscated/encoded PowerShell — each one is a meaningful finding by itself, regardless of how it was triggered or how long it ran.
MEDIUM
Short-lived processes and unusual parent/child spawns — a useful lead, but not proof, until corroborated by other alerts or the ML layer.

Key Finding

A single account — pbeesly — shows a complete, four-stage attack pattern on 2020-05-02 (UTC).

pbeesly — full event timeline, 02:55:57 – 03:21:32 UTC
Initial Access (ML-only) High severity Medium severity
STAGE 1 · 02:55:57 UTC

Initial Access — RTLO-spoofed screensaver

explorer.exe spawned a process disguised using a Unicode right-to-left override character (U+202E) — a filename-spoofing technique that makes a malicious .scr executable visually appear to end in a harmless extension. It ran from C:\ProgramData for 330.9 seconds.

This exact parent-child pairing occurred once across all 460 process-creation events. No rule was written to catch it — an Isolation Forest model flagged it anyway, based purely on how rare that pairing was.

STAGE 2 · 03:04 – 03:21 UTC (16.7 minutes)

Lateral Movement

pbeesly generated 30 logon events across three logon types (Network, Interactive, RDP) from four distinct source contexts, including 172.18.39.2, 10.0.1.4, and 10.0.1.6. 15 sessions were both remote and privilege-elevated, each triggering a High-severity Lateral Movement alert.

Within this window, PsExec64.exe was used four times to launch python.exe remotely on a host named NASHUA — with a plaintext domain password embedded directly in the command line, a common real-world consequence of process-command-line logging.

STAGE 3 · 03:02, 03:03, 03:16–03:17 UTC

Anti-Forensics

sdelete64.exe — a secure-deletion utility — ran 6 separate times, including once against the very RTLO-named file used for initial access. Severity is High regardless of how long each run took: a larger file legitimately takes longer to wipe, but the tool's presence is the signal, not its speed.

STAGE 4 · 03:21:31 – 03:21:32 UTC

Obfuscated Command Execution

Two PowerShell invocations closed out the session: one using -EncodedCommand, the second decoding a gzip-compressed, Base64-encoded loader that uses .NET reflection to run code entirely in memory — a fileless execution pattern that leaves no file on disk. Neither process has a recorded exit event, so without a presence-based rule, both would have been invisible to every duration-dependent check and the ML process model.
